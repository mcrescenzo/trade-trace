## Friction found

| id | surface | severity | disposition |
|----|---------|----------|-------------|
| **AX-051** | schema (`memory.link`) | low–med | **fixed** `2f1e170` |

**AX-051 — `memory.link` advertises enum-constrained fields as bare strings.**
`memory.link`'s description promises `source_kind`/`target_kind`/`edge_type` are
"validated against the documented enums", and runtime rejects bad values with
*self-documenting* errors that DO list the allowed set (verified live: bad
`edge_type` → lists the 7; bad `source_kind` → lists the 13). But `tool.schema
memory.link` advertised all three as bare `string` with no enum — it was
registered with **no explicit `json_schema`**, so the MCP schema auto-derived
from `example_minimal` (which carries only `memory_node/memory_node/supports`).
A bot reading the schema to build a valid call could not discover the 13
source/target kinds or 7 edge types without first triggering a
`VALIDATION_ERROR`. Its sibling `memory.reflect` already advertises the
*identical* 13-kind `target_kind` as a full enum via
`_MEMORY_REFLECT_SCHEMA` — `memory.link` was the lone outlier. Same
self-documenting-schema class as AX-010 (enum absent from advertised schema),
milder because the error recovers.

## Fixes made

- **AX-051** — `2f1e170` `fix(memory): memory.link schema advertises endpoint +
  edge_type enums`. Added `_MEMORY_LINK_SCHEMA` (enums sourced from the same
  `VALID_MEMORY_ENDPOINTS` / `EDGE_TYPES` runtime allowlists) and passed it as
  the explicit `json_schema`, mirroring `memory.reflect`. Additive only:
  runtime validation, accepted fields, and error messages unchanged. +1
  regression test (`test_memory_link_schema_advertises_endpoint_and_edge_enums`)
  asserting the advertised enums match the runtime allowlists exactly.
  **Effective next fresh session** (tool-code change; this session's live MCP
  still shows the old bare-string schema).
