### AX-045 — report.risk is unreachable from the MCP write surface
`report.risk` returned `sample_size:0`, `n_closed_total:0`,
`missing_risk_count:0`, **and empty `caveats`** on a journal with 49 decisions
and ~13 open paper positions — despite its docstring promising decisions
without declared risk are "counted in caveats so the agent can chase the gap."
Root cause is two-layered:
- Its missing-count is over **closed** decisions only (`risk.py:157`), and no
  paper position is ever closed (intentional position-lifecycle #1), so the
  gap is structurally invisible on this surface. `report.open_positions` *does*
  flag each row `missing_risk_budget` ("Record the risk you accepted at entry
  to recover this metric").
- But the field that advice points at — `declared_risk_amount` /
  `declared_risk_unit` — though **accepted** by `decision.add`
  (`decision.py:91`, threaded into the position's `initial_risk_amount` at
  `:228`) and depended on by `report.risk`, `report.opportunity`, and the
  open-positions caveat, was **absent from every advertised surface**: not in
  `x-decision-matrix` optional lists, not in the JSON-schema `properties`, not
  in the examples. A bot following the schema could never discover it, so
  `report.risk` was structurally always-empty for an MCP-only bot.

Same self-documenting-schema class as the fixed AX-030/032/035. **Fix
(903f9f4, additive, forbidden nowhere):** marked `declared_risk_amount` /
`declared_risk_unit` `O` on `paper_enter`/`actual_enter`/`add` in
`DECISION_MATRIX`; added both to `_DECISION_ADD_SCHEMA.properties` with a
description pointing at report.risk/opportunity; added them to the
`actual_enter` example. +2 regression tests (schema surfacing on all three
entry types; end-to-end that a `paper_enter` with `declared_risk_amount`
populates the position `initial_risk_amount` and clears `missing_risk_budget`).
Effective next fresh session (long-lived MCP server).

Residual (noted, not filed — backlog noise): `report.risk` still shows
`missing_risk_count:0` + empty caveats on an all-open risk-less journal because
R-multiples genuinely require closes; `report.open_positions` already serves
the "chase the gap" signal per-row.

### AX-046 — market.scan.promote final_check is wall-clock non-deterministic
Surfaced at the **full-suite gate**, not as a bot:
`tests/integration/test_market_scan_guided_e2e.py` failed
(`source_attached` step `'weak' == 'ok'`). Not caused by this run's change —
**proven** by reverting both touched source files to HEAD~1 and reproducing the
identical failure. Root cause: `market.scan.promote` builds `final_check` via
`journal.bundle.status` (`market_scan.py:443`) **without** forwarding a
deterministic clock or `stale_source_days`; `_stale_sources`
(`journal_bundle_status.py:379`) compares the source `captured_at` against
`datetime('now','-14 days')`. The test hardcoded source
`captured_at=2026-05-21T12:05Z`, which crossed the 14-day threshold at
2026-06-04T12:05Z — so it began failing on a clock tick. Fixed the test
(791f307) by making the bundle timestamps run-relative
(`datetime.now(UTC)`-derived, computed once at import for replay stability).
Filed the underlying **product** non-determinism as `trade-trace-efmq` (P3):
`promote` should thread `current_time`/`stale_source_days` into the terminal
`bundle.status` hop, mirroring `dry_run`'s existing deterministic-clock
affordance.
