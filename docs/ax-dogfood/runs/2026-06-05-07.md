## One-paragraph summary

Drove the canonical loop as a bot (bootstrap → scan → bind → snapshot → recall →
forecast → interpret_resolution → decision → calibration/coach → retain/recall →
cold probe), then spent the depth budget on the three genuinely-unprobed write
veins the run-06 handoff flagged. All three veins surfaced real friction: (1)
`forecast.interpret_resolution` silently no-ops a divergent second call (false
success), (2) `replay.evaluate_output` rejects fully-supplied candidate metadata
placed under the obvious key, and (3) the AX-051-style audit lead paid off —
`outcome.add`/`resolution.add` advertises its runtime-enum `status` as a bare
string. Fixed all three directly (additive, self-documenting; no contract-firewall
edits), each with a regression test. The replay artifact sub-suite
(`record`/`list`/`get`) and `replay.case_bundle` round-trip clean.

## Friction found

| id | surface | severity | finding |
|----|---------|----------|---------|
| AX-052 | tool (`forecast.interpret_resolution`) | medium | One-per-forecast collision returned the existing row with `ok:true` and **no signal**, even when the second call carried a *different* idempotency_key AND a *different* payload (market_contract vs original oracle_feed). A bot revising its resolution reading gets a false success and its revision is silently dropped. Distinct from intentional-design #7 (same key + diff payload → IDEMPOTENCY_CONFLICT); here a *different* key + diff payload silently no-ops via the forecast_id uniqueness guard. |
| AX-053 | tool (`replay.evaluate_output`) | medium | The `candidate_metadata` criterion reads metadata only from `candidate_output.metadata` (replay_evaluate.py:205), while the module is alias-tolerant everywhere else (case_id, citations). All 8 required fields supplied under the obvious `candidate_metadata` key (mirroring the bundle's own `candidate_metadata_required`) reported every field "Missing" — and there is no published `replay.candidate_output.v0` schema to disambiguate. The error can't distinguish "absent" from "wrong key". Isolated with a valid single-case bundle (bundle_contract/case_coverage/citation_coverage all pass). |
| AX-054 | schema (`outcome.add`/`resolve.record`/`resolution.add`) | medium | Registered with no explicit `json_schema`, so the MCP schema auto-derives from `example_minimal`: `status` advertised as a bare string though the runtime rejects out-of-enum values with a self-documenting error (10-value `_OUTCOME_STATUSES`), and the auto-score-gating `confidence` field appeared only in `example_rich`, not in `properties`. The AX-051 (memory.link) class; also the AX-030 confidence-discoverability residual. The handoff's explicit AX-051-style audit lead. |
