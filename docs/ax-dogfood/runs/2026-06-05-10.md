`redirect:forecast.add`). Two coupled defects: (1) the **shipped** doc `scoring.md`
(:105-107, :239-240, :193-224) and the loop's pinned `intentional-design.md` item #6
both name `forecast.supersede` as THE recovery path for `yes_label_ambiguous` /
`label_mismatch` — but an MCP-only bot has no such catalog tool; (2) the advisory
redirect is semantically **wrong** — `forecast.add` takes no `prior_forecast_id` and
performs no supersede, so following it would mint a SECOND live forecast on the same
market (double-counting calibration) while leaving the prior un-superseded. Same
doc/catalog-drift family as AX-058/`47tp` (folded `playbook.propose_version` vs
catalog-visible `record_adherence`). **Why filed, not fixed:** the resolution is a
catalog-shape / contract decision touching append-only forecast semantics
(contract-firewall-adjacent); editing `scoring.md` to name `forecast.add` would be
*incorrect* (it cannot supersede). Three design options are enumerated in the bead
(un-fold + explicit schema / keep-folded-fix-redirect + route `forecast.add` / accept
the deferral and update both docs to say so). Dedup: `bd search forecast.supersede /
supersede / yes_label / 'label correction' / 'folded catalog'` all empty; not covered
