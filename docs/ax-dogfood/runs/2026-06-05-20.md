### AX-068 — `market.refresh` hard-fails on every cache-miss (headline)

As a bot, I called `market.refresh` on my 3 standing open forecasts' markets to
check whether any had resolved (the natural open→resolved detection path).
**Every call** hard-errored:

```
event_type 'market.refreshed' is not registered in events_semantic_keys;
refusing to write to guard against silent contract drift (persistence.md §5.2)
```

Root cause (Phase C): `adapter_polymarket.py:361` emits
`event_type="market.refreshed"` when re-syncing a bound market from live Gamma,
but `market.refreshed` was never added to the default-deny `SEMANTIC_KEYS`
registry, so `events/log.py:224` raises `KeyError` and the whole call fails.
**Why it survived 67 prior friction items:** a freshly-bound market is inside
the open-market 1h adapter cache, so `market.refresh` short-circuits
(`cache_hit:true`) *before* the emit path — every prior run only refreshed via
`market.bind` re-enrich or `snapshot.fetch`, never `market.refresh` on an aged
row. The live failure needs the cold cache a long-lived server doesn't reach
mid-session.

Faithful fallback used as a bot: `snapshot.fetch` (works) confirmed all 3
markets still live (mids 0.505 / 0.19 / 0.34, all `accepting_orders=true`) — so
the 3 `resolve_due` obligations are the standing `resolution_at_missing`
false-positive class (`fe2f`); I did NOT fabricate resolutions.

## Fixes made

- **`4cb1307`** — `fix(events): register market.refreshed semantic key so
  market.refresh works (AX-068)`
  - Register `market.refreshed` in `SEMANTIC_KEYS`, mirroring `market.bound`
    (same payload shape).
  - Classify bucket-D (diagnostic) in `jsonl-replay-taxonomy.md` and
    `imports._DIAGNOSTIC_EVENT_TOOLS`: `market.refresh` re-fetches live and is
    not import-replayable, so a restored journal regenerates it on demand.
  - 3 regression tests: (1) a cache-miss `market.refresh` succeeds and writes
    the event; (2) `market.refreshed ∈ SEMANTIC_KEYS`; (3) a **source-wide
    guard** asserting every `event_type="x.y"` literal emitted in `src` is
    registered — catches the whole drift class (the existing alignment lints
    only walked the exporter map, not the emit sites, which is why this slipped
    through).
  - *Effective next fresh server session* (the live MCP server keeps old
    behavior until a cron `run.sh` fire / restart).
