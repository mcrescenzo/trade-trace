## Friction found

### AX-063 (NEW, fixed) — raw-IO admin writers ignore `_dry_run`; a "dry-run" mutates the live store

- **Surface:** tool / contract (`journal.config_set`, `journal.backup`,
  `journal.restore`, `model.import`).
- **Severity:** medium-high (a documented, uniformly-advertised safety affordance is
  silently false for these tools; `journal.restore` would overwrite the live DB).
- **Evidence:** `tool.schema.required_metadata.supports_dry_run` is computed as
  `reg.is_write` (`journal.py:340`), so **all** write tools advertise dry-run support
  via the generic `_dry_run` flag. The dispatcher implements it generically
  (`core.py:450-457`): it sets the request-scoped `DRY_RUN_FLAG`, which `UnitOfWork`
  reads in `_commit` and rolls back instead of committing
  (`unit_of_work.py:76-82`). But the admin writers bypass `UnitOfWork`:
  `journal.config_set` commits via a direct `sqlite3.connect` / `open_database`
  connection (`admin.py:603-627`); `journal.backup` copies files; `journal.restore`
  overwrites `$TRADE_TRACE_HOME`; `model.import` stages a model dir. Each gated only
  on `_confirm`, never on `DRY_RUN_FLAG`. **Confirmed live:** `journal.config_set`
  with `_confirm:true,_dry_run:true` returned `preview_only:false` and persisted
  `some.unknown.key=arbitrary` (and bumped `embeddings.provider.updated_at`) into the
  live axloop config table; I deleted the probe row afterward to leave the journal
  clean.
- **Disposition:** **fixed** — `5aed8eb`. Added a `_dry_run_active()` helper reading
  `DRY_RUN_FLAG` and short-circuited all four handlers to their existing
  `would_write`/`would_copy`/`would_restore` preview when dry-run is active. +4
  regression tests. The dry-run *contract* is unchanged (not a contract-firewall
  edit); the handlers now honor it. Effective in the live MCP **next fresh session**;
  verified now by unit tests.

### Secondary observation (not filed) — `journal.config_set` `value` enum absent from auto-derived `json_schema`

The `embeddings.provider` value is runtime-validated against the closed enum
`{none, local}`, and the **tool prose description** advertises it, but the
auto-derived `json_schema` shows `value` as a bare string. Unlike the AX-051/060
class this is (a) conditional on `key=embeddings.provider`, (b) already named in the
tool description a bot sees, and (c) the rejection error is fully self-documenting
(`details.allowed:["local","none"]`). Low severity, not a silent trap — recorded as
an observation, not filed.
